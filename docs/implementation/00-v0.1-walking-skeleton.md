# V0.1 Walking Skeleton

## 1. Purpose

The V0.1 Walking Skeleton is the earliest runnable evidence loop for Chtest. It exists to shorten the feedback cycle before the full V1 Minimum Demo is complete.

The full V1 demo remains `docs/fixtures/00-v1-demo-path.md`. V0.1 does not replace it. V0.1 proves that the platform spine works before Chtest invests in all review pages, metrics, Playwright, CI/CD Quality Center, and failure repair.

## 2. Product Question

V0.1 must answer one question:

```text
Can Chtest create project context, run a deterministic AI-like task, persist artifacts, execute a minimal pytest command, and generate an evidence-backed report record?
```

If the answer is yes, later slices can safely expand into requirement review, case review, AutomationDraft review, Playwright, CI/CD Quality Center, and richer metrics.

## 3. Scope

### In Scope

- Default single-user workspace and user.
- Project, Module, Repository, Environment, and TestCommand records.
- One ContextArtifact stored through the Artifact table.
- One deterministic mock AITask that records PromptVersion, SkillVersion, model, input, output, and context artifact ids.
- Artifact writes for AI task input/output and `context_manifest.json`.
- One minimal pytest execution through the TestRunner path or an equivalent early ToolInvocation stub.
- stdout, stderr, JUnit or parsed result artifact metadata.
- One minimal report JSON that cites evidence artifacts.
- A smoke command that proves the loop from seed/setup to report output.

### Out Of Scope

- Real LLM provider.
- Full RequirementReviewAgent behavior.
- Full GeneratedCaseCandidate review workflow.
- Full AutomationDraft approval UI.
- Playwright execution.
- CI/CD Quality Center.
- FailureAnalysis and AutomationRepairTask.
- Prompt/Skill A/B evaluation beyond deterministic schema validation.
- Enterprise collaboration, RBAC, RAG indexing, or MCP runtime integration.

## 4. Target Flow

```text
Seed default workspace/user
  -> create Checkout System project
  -> create module, repository, environment, test command
  -> create coupon-api-notes.md ContextArtifact
  -> run mock AITask with context_artifact_ids
  -> write input/output/schema/context_manifest artifacts
  -> execute sample pytest command
  -> capture stdout/stderr/JUnit or parsed result artifacts
  -> create minimal automation_execution report
  -> write evidence_manifest.json
```

## 5. Required Data

V0.1 uses the same concepts as V1, but only the minimum fields needed to prove the spine.

| Entity | Minimum Requirement |
|---|---|
| Workspace | Default personal workspace exists |
| User | Default user exists |
| Project | `Checkout System` exists |
| Module | `订单结算` or `Checkout` exists |
| Repository | Local sample repository path is allowlisted |
| Environment | `local` environment exists |
| TestCommand | `pytest tests -q --junitxml=artifacts/junit.xml` exists |
| Artifact | Context, AI task, runtime, stdout/stderr/JUnit, report artifacts can be persisted |
| AITask | Mock task records prompt, skill, model, input, output, and context ids |
| ToolInvocation | Test runner invocation records command, cwd, status, exit code, and artifacts |
| TestRun | Run records command, runner mode, workspace, status, parsed result, runtime metadata |
| Report | Minimal JSON report references evidence artifacts |

## 6. Required Artifacts

V0.1 must write or simulate these artifact records through the same Artifact contract used by V1:

```text
artifacts/projects/{project_id}/ai-tasks/{ai_task_id}/input.json
artifacts/projects/{project_id}/ai-tasks/{ai_task_id}/context_manifest.json
artifacts/projects/{project_id}/ai-tasks/{ai_task_id}/raw_output.json
artifacts/projects/{project_id}/ai-tasks/{ai_task_id}/parsed_output.json
artifacts/projects/{project_id}/test-runs/{test_run_id}/stdout.log
artifacts/projects/{project_id}/test-runs/{test_run_id}/stderr.log
artifacts/projects/{project_id}/test-runs/{test_run_id}/junit.xml or parsed_result.json
artifacts/projects/{project_id}/reports/{report_id}/report.json
artifacts/projects/{project_id}/reports/{report_id}/evidence_manifest.json
```

If JUnit parsing is not ready, `parsed_result.json` is acceptable for V0.1 only. The full V1 demo must return to the JUnit and artifact requirements in `docs/fixtures/00-v1-demo-path.md`.

## 7. Acceptance Criteria

V0.1 is accepted when all of these are true:

- `/health` and `/ready` pass.
- Seed/default workspace and user are available.
- Project Settings bootstrap can return project, module, repository, environment, and test command data.
- A ContextArtifact is persisted as an Artifact with `owner_entity_type=Project` and `owner_entity_id=project_id`.
- A mock AITask succeeds and includes an explicit `context_artifact_ids` list.
- The AITask artifact directory includes `context_manifest.json`.
- The test runner path executes or stubs one pytest command through the same state model that later TestRunner will use.
- The TestRun has `passed` or `failed` status based on execution evidence, not model text.
- A Report with `report_type=automation_execution` is created.
- The report's `evidence_manifest.json` references the test run artifacts.
- No generated asset bypasses review gates in production workflows; V0.1 shortcuts are explicitly marked as smoke-only.

## 8. Suggested Smoke Command

The final command can be implemented later, but the expected shape is:

```bash
cd backend && uv run pytest app/tests/e2e/test_v0_1_walking_skeleton.py -q
```

Expected result:

```text
1 passed
```

If the Docker runner is available by then, add a second smoke command:

```bash
docker compose -f deploy/docker-compose.yml up --build
```

Then verify:

```bash
curl -fsS http://localhost:8000/health
curl -fsS http://localhost:8000/ready
```

## 10. Relationship To V1 Minimum Demo

V0.1 is an early engineering checkpoint. V1 Minimum Demo is the product acceptance checkpoint.

V0.1 may use simplified or API-only surfaces. V1 must still satisfy:

- human review for generated cases;
- approval before AutomationDraft execution;
- full runner sandbox metadata;
- dependency and environment snapshots;
- evidence-backed report conclusion;
- user-visible workflow surfaces.

Do not mark V1 complete because V0.1 passes. Use V0.1 to reduce implementation risk and guide the next slices.
