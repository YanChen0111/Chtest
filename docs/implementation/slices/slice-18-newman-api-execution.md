# Slice 18: Newman API Execution Task Plan

## Goal

Add the first V2 runner expansion: local Newman API execution evidence.

This slice lets Chtest run an approved Newman API collection through the same
evidence loop used by pytest and Playwright: command allowlist, TestRun state,
stdout/stderr artifacts, machine-readable result artifacts, parsed results, and
reportable evidence. It must not become arbitrary shell execution, Postman
workspace parity, remote CI/CD integration, or a broad runner marketplace.

## Preconditions

- V1 is accepted as a local-first AI testing evidence workbench.
- TestCommand, ToolDefinition, ToolInvocation, TestRun, TestResult, Artifact,
  and Report patterns already exist.
- pytest and Playwright execution paths already prove the runner evidence loop.
- Tool execution remains allowlisted; users cannot run arbitrary shell commands.
- V2 scope options recommend Newman API execution as the first V2 slice.

## Product Value Answer

After this slice, a test engineer can run an approved Newman API collection
locally, inspect execution state, review stdout/stderr and Newman result
artifacts, see parsed API request/assertion outcomes, and include that evidence
in the existing Chtest workflow. This expands daily API testing coverage while
keeping Chtest focused on auditable local evidence instead of remote CI/CD or a
general automation platform.

## Non-goals

- No Postman workspace clone, collection editor, environment editor, mock
  server, monitor, cloud sync, or Postman account integration.
- No arbitrary shell execution; Newman must run through an allowlisted
  ToolDefinition/TestCommand boundary.
- No remote CI/CD provider control, PR comments, deployment automation, release
  automation, or external pipeline trigger.
- No broad runner marketplace or plugin installation flow.
- No JMeter, Appium, Cypress, Robot Framework, or performance/mobile runner
  implementation in this slice.
- No RAG runtime, MCP runtime dependency, RBAC, tenants, permissions, SSO, or
  enterprise audit.
- No secret manager; fixture credentials must stay deterministic and local.

## Slice Boundary

- Extend `TestCommand.command_type` to include `newman`.
- Add a Newman-specific allowlisted tool intent such as
  `newman_collection_run`.
- Store Newman stdout and stderr as existing text artifacts.
- Store machine-readable Newman output as a typed artifact, planned as
  `newman_json`.
- Optionally store Newman JUnit output as the existing `junit` artifact type
  when generated by the command.
- Store a `parsed_result` artifact summarizing collection, request, assertion,
  duration, and failure counts.
- Map parsed API assertions into TestResult rows with deterministic names such
  as `collection/folder/request::assertion`.
- Keep execution local and deterministic in tests by using fixtures or a fake
  Newman executable instead of network-dependent collections.
